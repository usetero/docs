## How Tero uses AI

Tero sends telemetry samples for classification. This powers:

- **Log event classification**: what each log event represents and whether it's valuable
- **Service enrichment**: context about what services do, based on their telemetry patterns
- **Waste identification**: which telemetry is debug noise, health checks, or otherwise low-value

See [Policies](/policies/overview) for how Tero turns classification into reviewable changes.

### What gets sent

Tero sends sample log lines, metric names, and span names. These samples include:

- Log message bodies and attributes
- Service names and descriptions
- Metric and span metadata

Tero processes samples in memory and discards them. Neither Tero nor the AI provider stores them.

### Usage scales with event diversity

AI usage depends on how many **unique event types** you have. Total telemetry volume doesn't affect it.

If you have 1 billion logs but they all match 1,000 distinct event types, Tero only needs to classify those 1,000 events. The other 999,999,000 logs match existing classifications and don't require AI calls.

Rough estimates:

| Environment | Unique event types | Initial analysis |
|-------------|-------------------|------------------|
| Small (10 services) | ~500-2,000 | ~50K-200K tokens |
| Medium (50 services) | ~2,000-10,000 | ~200K-1M tokens |
| Large (200+ services) | ~10,000-50,000 | ~1M-5M tokens |

After initial analysis, ongoing usage stays low. New event types become less common once Tero classifies your telemetry patterns.

<Note>
These are rough estimates. Actual usage depends on telemetry diversity. A service with verbose logging creates more unique event types than one with structured logs.
</Note>
