Workspaces let different teams manage the same data for different purposes. Security, observability, and compliance each get their own policies for the same catalog.

Most organizations don't need workspaces. One account, one set of policies, multiple teams. Workspaces become useful when teams have conflicting goals for the same telemetry.

## When to use workspaces

Your security team wants to keep all authentication logs with full context. Your observability team wants to sample them and drop the tokens. Meanwhile, your compliance team needs a 90-day archive of all of it.

Workspaces resolve the conflict: each team manages its own policies against the same catalog.

## How it works

The [Master Catalog](/master-catalog/overview) is shared across workspaces. Every workspace sees the same services, log events, metrics, and traces. Tero builds the catalog at the account level.

Policies are per-workspace. Each workspace evaluates the catalog and creates its own policies. The security workspace keeps auth logs while the observability workspace samples them; neither blocks the other.

## Enforcement

Workspaces map to destinations. Your routing determines how policies apply:

<Tabs>
  <Tab title="Separate destinations">
    Security logs go to your SIEM, observability logs go to Datadog. Each workspace's policies apply to its destination.

    ```mermaid
    flowchart LR
      Services --> Tero
      Tero --> Security[Security Workspace]
      Tero --> Observability[Observability Workspace]
      Security -->|security policies| SIEM
      Observability -->|observability policies| Datadog

      style Services fill:#262626,stroke:#262626,color:#fafafa
      style Tero fill:#10b981,stroke:#262626,color:#fff
      style Security fill:#0e2b22,stroke:#0e2b22,color:#fafafa
      style Observability fill:#0e2b22,stroke:#0e2b22,color:#fafafa
      style SIEM fill:#262626,stroke:#262626,color:#fafafa
      style Datadog fill:#632ca6,stroke:#262626,color:#fff
    ```

    This keeps separation clean: each team controls what reaches its destination.
  </Tab>
  <Tab title="Different indices">
    Both teams use Datadog, but security uses a dedicated index. Tero tags security-relevant logs, and Datadog's index filters route them.

    ```mermaid
    flowchart LR
      Services --> Tero
      Tero -->|tag + policies| Datadog
      Datadog -->|"tero.security:true"| SecurityIndex[Security Index]
      Datadog -->|everything else| MainIndex[Main Index]

      style Services fill:#262626,stroke:#262626,color:#fafafa
      style Tero fill:#10b981,stroke:#262626,color:#fff
      style Datadog fill:#632ca6,stroke:#262626,color:#fff
      style SecurityIndex fill:#0e2b22,stroke:#0e2b22,color:#fafafa
      style MainIndex fill:#0e2b22,stroke:#0e2b22,color:#fafafa
    ```

    The security workspace configures which logs get tagged as security-relevant. You set up a Datadog index with filter `tero.security:true` and order it before your main index. Security logs route there, everything else falls through.

    This can reduce security index costs because Datadog indexes only the logs your security team marks as security-relevant.
  </Tab>
  <Tab title="Same index">
    Both teams query the same data. Tero unions the policies before enforcement.

    ```mermaid
    flowchart LR
      Services --> Tero
      Tero --> Security[Security Workspace]
      Tero --> Observability[Observability Workspace]
      Security --> Union
      Observability --> Union
      Union -->|combined policies| Datadog

      style Services fill:#262626,stroke:#262626,color:#fafafa
      style Tero fill:#10b981,stroke:#262626,color:#fff
      style Security fill:#0e2b22,stroke:#0e2b22,color:#fafafa
      style Observability fill:#0e2b22,stroke:#0e2b22,color:#fafafa
      style Union fill:#10b981,stroke:#262626,color:#fff
      style Datadog fill:#632ca6,stroke:#262626,color:#fff
    ```

    The more permissive policy wins. If security wants to keep auth logs and observability wants to drop them, Tero keeps them.
  </Tab>
</Tabs>

## Creating workspaces

Create a workspace in **Settings → Workspaces**. Assign teams to it. Those teams manage policies for that workspace.

Each workspace has its own:
- Policy decisions and approvals
- Enforcement configuration
- Data quality metrics and SLOs

The catalog, services, and integrations remain shared at the account level.
