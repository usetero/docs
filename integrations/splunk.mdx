Connect Splunk when you want Tero to review Splunk log telemetry, build service and log-event context, and create supported provider actions through Splunk ingest controls.

## Connect Splunk

<Steps>
  <Step title="Run Tero setup">
    Open your terminal and run:

    ```bash
    tero
    ```

    Sign in when the setup flow opens your browser.
  </Step>

  <Step title="Select Splunk">
    Choose Splunk as the integration you want to connect.
  </Step>

  <Step title="Enter your Splunk URL">
    Enter your Splunk instance URL. For Splunk Cloud, use your Splunk Cloud hostname. For Splunk Enterprise, use your self-hosted URL.
  </Step>

  <Step title="Create an authentication token">
    In Splunk, go to **Settings > Tokens**. Create a token for a user with the role Tero should use. Copy the token into Tero setup.
  </Step>

  <Step title="Confirm analysis started">
    Tero validates the credentials and begins analyzing the connected log environment.
  </Step>
</Steps>

## What Splunk unlocks

Splunk gives Tero log evidence and provider context for:

- [Issues](/issues/overview) tied to Splunk log events and services
- [Log events](/master-catalog/log-events) grouped from Splunk logs
- Cost and compliance review when Tero can estimate affected volume and exposure
- Provider-side actions through Splunk ingest controls

## Provider actions

With write access, Tero can configure supported Splunk ingest actions for matching log telemetry.

Use provider actions when you want Splunk to apply the change. Use [Edge](/edge/overview), collector integrations, or repository workflows when you want the change to run outside Splunk.

## Permissions

Splunk uses capabilities to control access. Create a role with the capabilities Tero needs, or use an existing role.

| Role | What it gives Tero |
| ---- | ------------------ |
| **power** | Context only; you take action elsewhere. |
| **admin** | Context and provider actions. |
| **Custom** | Fine-grained access for the capabilities below. |

### Custom role capabilities

<AccordionGroup>
  <Accordion title="Context (read)">
    | Capability | What Tero can do |
    | ---------- | ---------------- |
    | `search` | Search logs for evidence and log-event grouping |
    | `list_inputs` | List data inputs |
    | `get_metadata` | Read index metadata |
  </Accordion>

  <Accordion title="Provider actions (write)">
    | Capability | What Tero can do |
    | ---------- | ---------------- |
    | `list_ingest_rulesets` | View ingest action rulesets |
    | `edit_ingest_rulesets` | Create and modify ingest action rulesets |
  </Accordion>
</AccordionGroup>

Start with read capabilities when you only want Tero to present evidence and recommendations. Add write capabilities when you want Tero to apply supported Splunk actions.
