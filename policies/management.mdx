Policy management covers what happens after Tero recommends or creates a policy: review state, versioning, sync, deployment, and impact.

## Policy state

The Tero control plane owns durable policy state. Tero presents that state so reviewers can inspect the policy, its source, linked issue, spec, deployments, and recent activity.

Current policy statuses:

| Status | Meaning |
| ------ | ------- |
| **Draft** | The policy exists for review or completion. |
| **Active** | The policy is enabled where it should run. |
| **Disabled** | The policy remains visible but is not intended to affect telemetry. |

## Policy source

Policy source tells reviewers where the policy came from. A policy may originate from a Tero recommendation, a provider import, a repository sync, or another configured workflow.

Use source together with the linked issue. Source explains origin. The linked issue explains evidence and intent.

## Versioning and repositories

Teams can use GitHub to review policy changes, open pull requests, create issues, and sync policies to repositories.

Repository sync is useful when engineers need policy review in the same workflow they use for service code. A synced policy can live in a central policy repository or near the service that owns the telemetry.

Use the [GitHub integration](/integrations/github) when you want repository-based review, audit history, or service-owner workflow.

## Runtime deployment

A policy has value when it reaches the place you chose to run it. Tero can show deployment state through policy detail and runtime surfaces such as Edge instances.

Common deployment targets include:

- Provider configuration, such as Datadog exclusion filters
- Edge instances running in your infrastructure
- Collector or pipeline integration paths
- Repository workflows that change instrumentation or configuration

Use the **Deployments** tab on policy detail to inspect where a policy runs and whether the runtime reports policy state.

## Provider inventory and imports

The Policies workspace can show Datadog exclusion filters alongside Tero policies. This helps you inspect customer-managed filters, evaluate import readiness, preview the policy branches Tero would create, and see imported policies.

Datadog filter conversion states can include **Proposal ready**, **Policy active**, **Policy verified**, **Rejected**, **Unsupported**, **Sample key required**, **Blocked**, and failure states.

## Impact

Policy management continues through impact. Review the linked issue, policy activity, and runtime state after deployment.

Use these checks:

- The issue moved from **Open** or **In progress** toward **Resolved**.
- The policy remains **Active** in the expected deployment target.
- Edge or provider deployment state reports no policy errors.
- Cost, compliance exposure, or matching volume moved in the expected direction.

If impact does not match the expected result, reopen the issue context and inspect evidence, spec, and runtime state together.
