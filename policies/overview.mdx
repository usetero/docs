Policies are data quality as code: atomic, portable statements about what's wrong with your data and how to fix it.

```yaml
id: checkout-api-drop-health-checks
name: Drop health check logs from checkout-api

log:
  match:
    - resource_attribute: service.name
      exact: checkout-api
    - log_field: body
      regex: "GET /health"
  keep: none
```

That's a policy. It does one thing and states the telemetry it changes. You do not need pipeline context or a dependency graph to review it.

## Why policies

Traditional pipeline configs often start clean and grow into thousands of lines with shared state, ordering constraints, and hidden dependencies. You need broad config knowledge to change one part safely. Automation has the same problem because each change can affect later stages.

We saw this repeatedly while building [Vector](https://vector.dev). Across large deployments, the same failure mode appeared: configs became hard to maintain because the architecture coupled unrelated changes.

Policies avoid that coupling. Each one is independent: no ordering dependencies, no shared state. Ten thousand policies execute as fast as ten. You can add or remove a policy without tracing the rest of the policy set. That independence keeps review manageable at scale.

The same independence makes policies portable. A policy can enforce at the [edge](/policies/enforcement/edge), push to your [provider's API](/policies/enforcement/provider), or sync to a pipeline. The policy is the source of truth. Enforcement translates it into the target runtime.

## How policies are created

Two sources:

**Foundation policies** are broad rules you write. Drop debug logs in production. Redact CVV fields. Sample health checks to 10%. These apply across services and rarely change.

**Generated policies** come from Tero. The [Master Catalog](/master-catalog) identifies your log events, understands what they mean, and generates specific policies for each. You might have hundreds: one per log event type that needs attention.

You review policies by [category](/policies/categories). Categories focus on evidence Tero can show: duplicate fields, malformed data, health checks, and PII in fields with sensitive names or values. Each review includes examples from your telemetry so engineers can decide whether to approve the change.

## What policies can do

Policies target logs or metrics and specify an action:

**Filter** - Control what gets kept:
- `keep: all` - keep everything (default)
- `keep: none` - drop it
- `keep: 50%` - sample to a percentage
- `keep: 100/s` - rate limit

**Transform** - Modify what survives:
- Remove fields
- Redact sensitive values
- Rename attributes
- Add metadata

See the full [policy spec](https://github.com/usetero/policy) for details.

## Where policies live

Policies aren't hidden in a vendor UI. They sync to where your team can see them:

- **Tero**: review and approve in the interface
- **Central repo**: platform team maintains a policies repository  
- **Service repos**: `.tero/` folder in each service, so engineers see what's happening to their data

If Tero drops or transforms data from a service, the policy is visible in the code for engineers to review.

See [management](/policies/management) for details on syncing and version control.
